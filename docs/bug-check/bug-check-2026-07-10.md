## 问题列表（按严重程度排序）

### 一、严重问题

#### BUG-01 · JWT 验签"算法白名单"形同虚设

- **文件**：[JwtVerifyTool.tsx](file:///e:/work/auto-website/src/components/JwtVerifyTool.tsx) 第 152 行；[jwtVerify.ts](file:///e:/work/auto-website/src/utils/jwtVerify.ts) 第 515 行
- **严重程度**：严重（安全）
- **问题描述**：组件中 `expectedAlg` 取自 token 自身解析的 alg：

  ```typescript
  // JwtVerifyTool.tsx 第 152 行
  const expectedAlg = enforceAlgWhitelist ? detected.alg : undefined;
  const result = await verifyJwt(tokenInput, keyInput, keyFormat, expectedAlg);
  ```

  `detected.alg` 来自 `detectAlg(tokenInput)`，即 token Header 段。而 `verifyJwt` 内部（第 481 行）的 `alg` 同样从同一 token 的 Header 解析。因此 `expectedAlg === alg` **恒成立**，白名单校验 `if (expectedAlg && expectedAlg !== alg)`（第 515 行）永远不会进入失败分支。

  无论"强制算法白名单"开关开或关行为完全相同，**无法防御算法混淆攻击**（如攻击者将 RS256 改为 HS256，用 RSA 公钥作为 HMAC 密钥验签）。`alg=none` 虽被后续逻辑拒绝（第 536-557 行），但白名单的设计意图——防止 alg 被篡改——完全未实现。
- **修复建议**：白名单应由调用方指定固定的允许算法集合（如仅 `HS256`/`RS256`），而非从 token 自身取 alg。改为在 UI 中提供"期望算法"下拉框，将用户选择的值作为 `expectedAlg` 传入。

### 二、高危问题

#### BUG-02 · JSON Schema `pattern` / `patternProperties` ReDoS 风险

- **文件**：[jsonSchema.ts](file:///e:/work/auto-website/src/utils/jsonSchema.ts) 第 207-216 行、第 302-312 行
- **严重程度**：高（安全 / ReDoS）
- **问题描述**：`pattern` 与 `patternProperties` 关键字直接用用户输入构造正则并执行 `new RegExp(sch.pattern).test(instance)`。若用户粘贴含恶意回溯型正则的 Schema（如 `{"pattern": "(a+)+"}`）配合长字符串数据，`RegExp.test()` 会导致指数级回溯，浏览器卡死。`JsonSchemaTool.tsx` 使用 `useMemo` 实时校验，每次输入都触发。
- **修复建议**：① 对 pattern 字符串长度设限；② 对被匹配字符串长度设上限；③ 在 Web Worker 中执行带超时的校验；④ 限制正则标志（如 `dotAll`）减少回溯。

#### BUG-03 · JSONPath `=~` 正则匹配 ReDoS 风险

- **文件**：[jsonPath.ts](file:///e:/work/auto-website/src/utils/jsonPath.ts) 第 750-758 行
- **严重程度**：高（安全 / ReDoS）
- **问题描述**：过滤表达式 `=~` 直接用用户输入构造正则 `new RegExp(rightVal).test(leftVal)`。恶意 JSONPath（如 `$..book[?(@.title =~ "(a+)+")]`）配合长字符串可导致 ReDoS。`JsonPathTool.tsx` 用 `useMemo` 实时查询，每次输入触发。
- **修复建议**：对正则字符串与被匹配字符串长度设限；考虑在 Web Worker 中带超时执行。

#### BUG-04 · JWE PBES2 迭代次数 `p2c` 无上限，存在 DoS 风险

- **文件**：[jwe.ts](file:///e:/work/auto-website/src/utils/jwe.ts) 第 655-660 行、第 679-688 行
- **严重程度**：高（安全 / DoS）
- **问题描述**：PBES2 算法从 JWE Protected Header 提取 `p2c`（迭代次数），仅校验 `p2c > 0`，无上限。随后直接传入 PBKDF2 `deriveBits` 的 `iterations` 参数。恶意 JWE 可设 `p2c` 为极大值（如 `1000000000`），导致 PBKDF2 计算耗尽 CPU，浏览器无响应。
- **修复建议**：增加 `p2c` 上限校验（如 `p2c > 10000000` 时拒绝），参考 RFC 7518 Section 4.8.1.2。

#### BUG-05 · htmlFormatter `preserveWhitespace` 模式下文本内容重复输出

- **文件**：[htmlFormatter.ts](file:///e:/work/auto-website/src/utils/htmlFormatter.ts) 第 226 行
- **严重程度**：高（功能错误）
- **问题描述**：序列化逻辑为：

  ```typescript
  lines.push(indent + text.trim() + (opts.preserveWhitespace ? text : ''));
  ```

  当 `preserveWhitespace` 为 `true` 时，输出为 `text.trim() + text`，即先输出去空白文本再追加完整文本，**内容重复**。例如 `" Hello World "` 输出为 `"Hello World Hello World "`。
- **修复建议**：根据 `preserveWhitespace` 决定是否保留原始空白，而非追加：

  ```typescript
  const display = opts.preserveWhitespace ? (child.textContent || '') : text.trim();
  if (display !== '') {
    lines.push(indent + display);
  }
  ```

#### BUG-06 · JsonToXmlTool / XmlToJsonTool 在 JSX 中使用 `class` 而非 `className`

- **文件**：[JsonToXmlTool.tsx](file:///e:/work/auto-website/src/components/JsonToXmlTool.tsx)（39 处）；[XmlToJsonTool.tsx](file:///e:/work/auto-website/src/components/XmlToJsonTool.tsx)（45 处）
- **严重程度**：高（代码规范 / 潜在样式失效）
- **问题描述**：两个组件在 JSX 中全部使用 `class=` 而非 React 标准的 `className=`，且 `for=` 应为 `htmlFor=`。React 18 虽会将未知属性透传到 DOM（样式可能仍生效），但会输出 dev 警告 `Invalid DOM property 'class'. Did you mean 'className'?`，且不符合 React 规范。项目中其他组件均使用 `className`，这两个是异常值。
- **修复建议**：将全文 `class=` 替换为 `className=`，`for=` 替换为 `htmlFor=`，`spellcheck` 替换为 `spellCheck`。

  > **注**：`SqlTool.tsx`（1 处）与 `MarkdownTool.tsx`（4 处）中的 `class=` 位于**模板字符串构造的 HTML**（用于 `dangerouslySetInnerHTML`），属于原生 HTML，**无需修改**。

### 三、中危问题

#### BUG-07 · RSA PKCS#1 / EC SEC1 格式私钥导入失败

- **文件**：[jwtSign.ts](file:///e:/work/auto-website/src/utils/jwtSign.ts) 第 374-387 行、第 468-481 行
- **严重程度**：中（功能）
- **问题描述**：`importRsaPrivateKey` 统一用 `'pkcs8'` 格式导入，但 `pemToDer`（第 267-281 行）正则同时匹配 `BEGIN RSA PRIVATE KEY`（PKCS#1）和 `BEGIN PRIVATE KEY`（PKCS#8）。PKCS#1 格式 DER 结构与 PKCS#8 不同，用 `'pkcs8'` 导入会抛异常。EC 私钥同理（SEC1 格式 `BEGIN EC PRIVATE KEY`）。
- **修复建议**：参考 `jwtVerify.ts` 中公钥的降级策略（先 SPKI 再 PKCS#1），私钥也实现降级：先 `'pkcs8'`，失败后 `'pkcs1'`（RSA）或 `'sec1'`（EC）。

#### BUG-08 · JweTool 渲染时 `base64urlDecode` 未捕获异常导致组件崩溃

- **文件**：[JweTool.tsx](file:///e:/work/auto-website/src/components/JweTool.tsx) 第 576 行

- **修复建议**：将第 576 行包裹在 try-catch 中，异常时返回空 `Uint8Array`。

#### BUG-09 · JSONPath `..[expr]` 递归下降后接括号解析错误

- **文件**：[jsonPath.ts](file:///e:/work/auto-website/src/utils/jsonPath.ts) 第 370-382 行
- **严重程度**：中（功能）
- **问题描述**：`parseSegment` 中 `DOTDOT` 后接 `LBRACKET` 时调用 `parseBracket(ctx)` 消费整个 `[...]`，再 `ctx.pos--` 仅回退到 `RBRACKET` 位置。外层循环再次 `parseSegment` 时 `peek` 返回 `RBRACKET`，不匹配任何合法段，抛出错误。导致 `$..[0]`、`$..[?(...)]` 等合法表达式解析失败。
- **修复建议**：移除 `parseBracket` 调用和 `ctx.pos--` 回退，直接返回 `{ kind: 'recursive', name: null }`，让外层循环自然解析后续 `[expr]` 段。

#### BUG-10 · JSONPath 多键 `['a','b']` 解析为永远不匹配的 placeholder

#### BUG-17 · sql.ts `formatSql` 强制追加末尾分号可能改变语义

## 整体评价

项目代码质量整体较高：
- 无 `eval` / `new Function`，用户内容主要通过 JSX 渲染（自动转义）
- Web Crypto API 使用基本正确（AES-GCM/CBC/CTR、PBKDF2、HMAC、ECDSA、ECDH、AES-KW、RSA-OAEP）
- 无 Node 专属 API，SSR 安全
- 无障碍基础完善（skip-link、`sr-only`、ARIA 属性、`lang="zh-CN"`）
- `robots.txt`、`og-image.svg`、联系邮箱均已使用正确域名

主要风险集中在三方面：① 加密工具的安全防御未真正生效（BUG-01）；② 用户输入构造正则的 ReDoS 风险（BUG-02/03/15）；③ 占位域名清理不彻底（依赖运行时兜底而非源头修复）。建议优先处理安全类问题。
